# 搜索扩容策略

本文件用于规定 llm-wiki 何时需要从纯 `index-first` 导航进入更强的本地搜索阶段，以及进入时应优先比较哪些方案。

## 默认原则

- 在 20–100 条来源规模内，优先使用 `wiki/_meta/index.md` 与 `state/page_graph.json`
- 搜索增强不是 MVP 前提，而是规模驱动的后续扩展
- 只有在真实试运行确认召回或效率不足时，才引入新的搜索层

## 触发门槛

满足以下任一情况时，可以启动搜索扩容评估：

- 相关页面需要人工翻页筛选，agent query 前的候选页面经常明显偏离主题
- Wiki 页面规模超过 100 页后，单纯依赖 index 已难以快速定位信息
- 试运行中出现稳定的多跳问题，必须手动扫读大量页面才能回答
- 团队明确需要关键词搜索、语义搜索或混合搜索，而现有状态文件不足以支撑

## 第一阶段方案

### 方案 A：继续强化 index + page graph

适用条件：
- 规模仍不大
- 主要问题在于索引摘要不足或链接关系稀疏

建议动作：
- 强化摘要质量
- 提高页面间链接密度
- 让 agent query 前的候选召回更充分利用 `state/page_graph.json`

### 方案 B：接入本地全文搜索

适用条件：
- 问题以关键词检索为主
- 需要快速定位页面，而不是复杂语义召回

建议动作：
- 评估轻量全文搜索器
- 保持返回结果仍以 repo-relative 页面路径表达

### 方案 C：接入 qmd 或混合搜索

适用条件：
- 来源与页面规模继续增长
- 问题经常跨主题、跨页面，需要更强召回

建议动作：
- 评估 qmd 的本地 CLI 或 MCP 形态
- 继续保持 `wiki/` 为知识真源，搜索层只提供召回，不负责写入

## 不应做的事

- 在未验证真实召回瓶颈前，提前引入向量数据库
- 让搜索层成为新的事实源
- 让 agent 绕过 wiki，重新退回对 `raw/` 的一次性临时检索

## 评估记录建议

每次进入搜索扩容评估时，至少记录：

- 当前来源数
- 当前 wiki 页数
- 最常见的问答失败类型
- `index-first` 方案的具体不足
- 候选搜索方案的接入成本、维护成本与回滚方式
